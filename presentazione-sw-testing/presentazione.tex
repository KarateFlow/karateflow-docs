\begin{frame}{3. L'applicazione (Ambiente di Staging)}
  L'applicazione web è interamente \textbf{hostata su un server Raspberry Pi}, fungendo da ambiente di Staging reale e accessibile dall'esterno.
  \vspace{0.2cm}
  
  \begin{tcolorbox}[colback=karateGray, colframe=karateDark, title=Accesso alla Piattaforma]
    \textbf{Link:} \href{https://staging.karate-flow.com}{\textbf{staging.karate-flow.com}}
  \end{tcolorbox}
  \vspace{0.2cm}
  
  \textbf{Principali funzionalità testabili in ambiente di staging:}
  \begin{itemize}
    \item \textbf{Gestione Anagrafica:} È possibile registrare e aggiungere un nuovo atleta al sistema.
    \item \textbf{Registrazione Misurazioni:} Si possono creare e associare specifici test (tecnici o atletici) ai profili registrati.
    \item \textbf{Analisi e Confronti:} Aggiungendo \textbf{almeno due test} per un singolo atleta, l'applicazione sblocca la funzionalità di confronto, permettendo di generare un report visivo per analizzare l'andamento delle misurazioni nel tempo.
  \end{itemize}
\end{frame}

\begin{frame}{4. Contesto e Obiettivi del Corso di Software Testing}
  Il progetto KarateFlow, sviluppato inizialmente come evoluzione per il corso di \textbf{SPME}, viene integrato con una robusta suite di test e automazione per il corso di \textbf{Software Testing}.
  \vspace{0.4cm}
  
  I tre obiettivi principali del progetto di Software Testing sono:
  \vspace{0.2cm}
  \begin{enumerate}
    \item \textbf{Integrazione di Testing Avanzato:} Mutation Testing, Performance & Load Testing, Automated E2E Testing.
    \item \textbf{Aumento della Coverage e Riduzione del Debito Tecnico:} Incremento della copertura dei branch ed eliminazione sistematica delle anomalie individuate dai linter e SonarQube.
    \item \textbf{Integrazione di GitHub Actions:} Realizzazione di pipeline in cloud e confronto con l'attuale strumento di CI/CD locale \textbf{Jenkins}.
  \end{enumerate}
\end{frame}

\begin{frame}{4.5 Esecuzione del progetto in locale (Docker-based)}
  L'intero ambiente di sviluppo locale è stato containerizzato, semplificando il setup ed eliminando la necessità di installare toolchain locali (Node, Maven, CLI).
  \vspace{0.2cm}

  L'avvio dell'applicazione richiede l'esecuzione sequenziale di due \texttt{docker-compose}:
  \begin{enumerate}
    \item \textbf{Backend (Primo step):} Dalla cartella \texttt{karateflow-be}, \texttt{docker-compose up} avvia sia \textbf{MongoDB} che l'API in \textbf{Spring Boot}, montando il codice sorgente per il live-reload e abilitando il debugging remoto (porta 5005).
    \item \textbf{Frontend (Secondo step):} Successivamente, dalla cartella \texttt{karateflow-fe}, \texttt{docker-compose up} avvia l'app \textbf{Angular} tramite un container Node dedicato allo sviluppo, configurato per rilevare istantaneamente le modifiche al codice (polling).
  \end{enumerate}
  \vspace{0.2cm}
  \textit{Nota: Anche l'ambiente CI/CD (Jenkins) può essere avviato opzionalmente tramite Docker dalla cartella \texttt{karateflow-infra}.}
\end{frame}

\begin{frame}{4.6 Testing Suite (Backend \& Frontend)}
  La strategia di Quality Assurance si divide in due stack tecnologici distinti:
  \vspace{0.2cm}
  
  \textbf{Backend (Spring Boot / Java)}
  \begin{itemize}
    \item \textbf{Unit \& Integration Testing:} Sviluppati con \textbf{JUnit 5} e \textbf{Spring Boot Test} per testare unitariamente la logica e in integrazione le API REST.
    \item \textbf{Testcontainers:} Utilizzato per l'avvio di container effimeri di \textbf{MongoDB} durante i test di integrazione, garantendo un I/O reale rispetto ai database mockati.
    \item \textbf{Code Quality:} Linting del codice tramite \textbf{PMD} e code coverage misurata tramite \textbf{JaCoCo}.
  \end{itemize}
  
  \vspace{0.2cm}
  \textbf{Frontend (Angular / TypeScript)}
  \begin{itemize}
    \item \textbf{Unit Testing:} Adozione di \textbf{Vitest}, test runner moderno con report agganciati a SonarQube.
    \item \textbf{Code Quality:} Analisi statica tramite \textbf{ESLint}.
  \end{itemize}
\end{frame}

\begin{frame}{8. Automated E2E Testing: Strumenti e Approccio}
  Per il testing End-to-End dell'applicazione web si è adottato un approccio \textbf{Behavior-Driven Development (BDD)} basato sulle seguenti tecnologie:
  \vspace{0.3cm}
  \begin{itemize}
    \item \textbf{Cucumber:} Utilizzato per scrivere gli scenari di test in linguaggio naturale (Gherkin), garantendo una chiara definizione dei requisiti di business e un'elevata leggibilità.
    \item \textbf{Playwright:} Framework moderno impiegato "sotto il cofano" per l'automazione cross-browser. Consente di simulare in modo affidabile le interazioni reali dell'utente (es. compilazione dei form, click, navigazione).
    \item \textbf{Esecuzione:} La suite di test viene lanciata tramite il comando \texttt{npm run test:e2e} (che orchestra Playwright tramite cucumber-js).
    \item \textbf{Scenari scelti}: Ho implementato correttamente due scenari: Creazione dei test e Gestione Atleti.
  \end{itemize}
\end{frame}

\begin{frame}{11. DevOps: Github Actions vs Jenkins}
  \begin{itemize}
    \item \textbf{Self-hosted vs GitHub-hosted runners:} Si è provato a usare runner self-hosted per GitHub Actions sul Raspberry Pi, ma saturavano le risorse impiegando troppo tempo. Si è preferito usare i server ospitati da GitHub.
    \item \textbf{Performance:} \textbf{Jenkins} (in esecuzione sulla macchina di sviluppo) risulta \textbf{leggermente più veloce} rispetto ai workflow eseguiti sui server GitHub.
    \item \textbf{Tempi di esecuzione:} Entrambe le soluzioni riescono comunque a concludere i task in \textbf{meno di un minuto}.
    \item \textbf{Trigger:} Jenkins viene avviato manualmente o con pooling mentre i workflow GHA si avviano manualmente ad ogni PR. 
  \end{itemize}
\end{frame}
